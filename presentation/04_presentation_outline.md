# SyncSpace — Stakeholder Presentation Outline

> **Audience:** Corporate sponsors, finance leads, IT governance.
> **Goal:** Convince decision-makers that SyncSpace enforces the agreed B2B workflow, ring-fences financial truth, and rests on a database that protects data integrity by construction.
> **Deck length:** 14 slides + appendix · ~25 minutes including Q&A.

## Slide 2 — Why a B2B PM tool, why now

**Title:** _The cost of fragmented project tracking_
**Layout:** Two columns. Left: pain points icon list. Right: a quantified "today vs. tomorrow" comparison card.
**Talking points:**
- Today: project notes live in spreadsheets, Slack, email; quotations drift from what is actually invoiced.
- Today: there is no programmatic enforcement that a Client cannot edit a WorkingStaff's task or vice-versa.
- Tomorrow with SyncSpace: every state change is captured in a constrained PostgreSQL row; every persona sees only the columns they own.
- Result: faster billing cycles, lower reconciliation overhead, audit-grade history out of the box.

## Slide 3 — Roles & access model

**Title:** _Three roles, one access matrix, zero public signup_
**Layout:** Horizontal "swim-lane" diagram with three figures (Manager / Client / WorkingStaff) and arrows showing what each can read & write.
**Talking points:**
- **Manager** is the only role that can provision new Companies and Users — the `users` table is enforced by the `user_role` ENUM at the database level, so even a misbehaving API call cannot insert an invalid role.
- **Client** can submit Project Proposals and review submitted work; they cannot modify task structure.
- **WorkingStaff** executes assigned tasks and submits deliverables; they have no access to financial tables.
- Public signup is intentionally absent — onboarding is a deliberate, audited act by a Manager.

## Slide 4 — Workflow at a glance (DFD Level 1)

**Title:** _From proposal → execution → invoice in one diagram_
**Layout:** Embed the prototype's Mermaid DFD (`prototype/syncspace.html` → "Workflow DFD"). Annotate the five processes 1.0 → 5.0 with timing benchmarks (e.g. "median proposal-to-approval = 3 days").
**Talking points:**
- Process 1.0 — Client submits a proposal; row inserted into `projects` with status `Proposed`.
- Process 2.0 — Manager approves and decomposes into tasks/subtasks; status moves to `Approved` then `InProgress`.
- Process 3.0 — Staff executes; subtasks are checked off and `actual_completed_at` is logged.
- Process 4.0 — Client reviews submissions; `work_submissions.review_status` is set.
- Process 5.0 — Manager issues an invoice; `invoices.status` lifecycle: `Draft` → `Issued` → `Paid` (or `Overdue`/`Void`).
- Key point: each process writes to *exactly one* data store, so audit-trail provenance is unambiguous.

## Slide 5 — The database is the contract (ERD)

**Title:** _Seven tables, four ENUMs, one source of truth_
**Layout:** Embed the HTML ERD from the prototype (`Database ERD` view) or screenshot it. Highlight in colour:
- 🟡 yellow boxes = Primary Keys
- 🔵 cyan boxes = Foreign / Unique Keys
- 🟢 green = monetary fields (`NUMERIC(14,2)`)

**Talking points:**
- All identifiers are **UUIDs** generated by `pgcrypto` — no enumerable integer IDs leaking row counts to the outside world.
- All money is stored as `NUMERIC(14,2)` (not floating point) — billing reports cannot drift by a cent.
- All states are **ENUMs**: `user_role`, `project_status`, `task_status`, `invoice_status`. Renaming a state is an explicit `ALTER TYPE` migration, never a typo.
- Cross-row business rules are enforced by `CHECK` constraints — e.g. `planned_end_date >= planned_start_date`, `paid_at` only allowed when `status = 'Paid'`. Bad rows cannot exist.
- Cascading rules are deliberate: deleting a project removes its tasks and invoices (`ON DELETE CASCADE`), but you cannot delete a Company that still has users (`ON DELETE RESTRICT`). The schema *prevents* destructive accidents.

## Slide 6 — Manager experience (Inputs)

**Title:** _The Manager controls intake, structure, and assignment_
**Layout:** Three side-by-side mockups taken from the prototype: Provisioning, New Project, Task Delegator.
**Talking points:**
- **Provisioning form** — single screen creates a Company row, then provisions Users into it; password is set by the Manager and rotated at first login.
- **Project form** — captures the **Quotation value** (the contractual ceiling) and **Planned Start/End** dates; database-level checks prevent illogical date ranges.
- **Task Delegator** — splits a project into tasks and subtasks; assignments are FKs to the `users` table so deleting a staff member safely sets `assigned_to = NULL` rather than orphaning data.

## Slide 8 — WorkingStaff experience (Execution & Submission)

**Title:** _A focused workspace, zero context-switching_
**Layout:** Show the Kanban board view alongside the Task Execution checklist card.
**Talking points:**
- Personal **Kanban** board filtered by `tasks.assigned_to = current_user.id` — no global noise.
- Each task surfaces its **subtask checklist**; ticking a subtask logs `done_at` and `done_by`.
- The **Submission form** writes to `work_submissions` with a delivery URL; the Client receives a review request automatically.
- The DB enforces `subtasks_done_consistency_chk`: a row cannot say "done = true" without a `done_at` date — staff cannot fudge half-completed work.

## Slide 9 — Financial transparency (Quotation vs. Invoice)

**Title:** _Every dollar promised is reconciled to a dollar billed_
**Layout:** Side-by-side: Manager **Financial Ledger** view (per-project bar showing Quotation vs. Invoiced %) and the **Client Invoice Center** (paid / issued / overdue KPIs).
**Talking points:**
- A project is the parent of a **1-to-many** relationship with `invoices` — partial milestone billing is first-class, not an afterthought.
- "Outstanding" = `quotation_amount − Σ(amount_issued where status NOT IN ('Draft','Void'))`.
- Invoice statuses follow a state machine; `paid_at` cannot be set unless status is `Paid` (DB-level CHECK).
- Both the Manager dashboard and the Client billing center read from the **same `invoices` table** — there is no possibility of the two sides seeing different numbers.

## Slide 10 — Schedule integrity (Expected vs. Actual)

**Title:** _Variance is a metric, not a surprise_
**Layout:** Embed a screenshot of the **Schedule Variance Report** with its colour-coded pills (green = early, blue = on-time, yellow/red = late).
**Talking points:**
- Every project carries **four dates**: planned_start, planned_end, actual_start, actual_end.
- Every task adds two more: expected_due and actual_completed_at.
- Variance is computed at presentation time — no nightly batch job, no drift between dashboard and ledger.
- Managers see project-level *and* task-level variance side by side; the same data feeds the Client tracker so there are no "surprise" delays.

## Slide 11 — MIS reports for executive review

**Title:** _Two reports that close the C-suite review meeting_
**Layout:** Split slide — left: Schedule Variance summary, right: Financial Yield. Both feature large summary KPIs at top.
**Talking points:**
- **Schedule Variance Report** — answers "are we delivering when we said we would?" Color-coded by traffic-light thresholds.
- **Financial Yield Report** — per-client view of total **Quoted** vs. **Invoiced** vs. **Realized (Paid)**.
- Yield % = invoiced / quoted; Realized % = paid / invoiced — together they identify *both* under-billing risk and slow-paying clients.
- Reports are **read-only views over the same tables** — there is never a separate "reporting database" to keep in sync.

## Slide 12 — Performance management

**Title:** _Velocity by person, not by guesswork_
**Layout:** Screenshot of the Manager **Performance Grid**: name, tasks assigned, tasks approved, subtasks done, completion %, average lead time.
**Talking points:**
- Velocity isn't an opinion — it is a SQL aggregation: `COUNT(*) FILTER (WHERE status='Approved')` over each staff member.
- Average lead time = `actual_completed_at − actual_start` per task; surfaces bottlenecks before they slip into Schedule Variance.
- Privacy: WorkingStaff see only their own velocity in the Kanban view; aggregated cross-staff comparison is gated to the Manager role.

## Slide 13 — How PgAdmin secures the back end

**Title:** _Operational safeguards in PgAdmin & PostgreSQL_
**Layout:** Three-column safety matrix.
**Talking points:**
- **Role separation:** the application connects with a least-privilege role (`syncspace_app`) — *not* `postgres`. Even a compromised application connection cannot drop tables.
- **Schema as code:** the entire schema lives in `db/02_schema_and_seed.sql`, version-controlled. Rebuilding a fresh environment is one paste + F5.
- **Idempotent migrations:** the script's `DROP IF EXISTS` block in correct dependency order means re-running on a soiled database is safe.
- **Backups:** PgAdmin's built-in `pg_dump` integration runs nightly; ENUM definitions and FKs are part of the dump, so a restore is a *complete* replay, not a partial one.
- **Audit:** `created_at` / `updated_at` columns plus row-level INSERT triggers (planned for Phase 2) deliver a tamper-evident trail.

## Appendix A — Data dictionary cheat-sheet

| Table | Owner role for writes | Notable constraints |
|---|---|---|
| `companies` | Manager | Email regex; unique legal_name |
| `users` | Manager | ENUM-typed role; FK to companies (RESTRICT on delete) |
| `projects` | Manager (Client may insert as `Proposed`) | Quotation ≥ 0; planned_end ≥ planned_start; actual dates monotone |
| `tasks` | Manager | Priority 1–5; expected_due ≥ expected_start |
| `subtasks` | WorkingStaff | done_at consistency CHECK with is_done |
| `invoices` | Manager | amount_issued > 0; due ≥ issue; paid_at gated by status |
| `work_submissions` | WorkingStaff | review_status reuses task_status ENUM |

## Appendix B — Question-handling crib sheet

- *"What if a Client tries to edit a task?"* — Application layer rejects on role mismatch; even if it didn't, the Client connection role lacks `UPDATE` privileges on the `tasks` table.
- *"How do you prevent double-billing?"* — Each `invoice_number` is a unique key; the Financial Ledger sums `amount_issued` and surfaces over-billing visually if `Σ(invoiced) > quotation_amount`.
- *"What if a project is cancelled mid-flight?"* — Status moves to `Cancelled`; tasks remain for audit. Optional cleanup uses `ON DELETE CASCADE` only when the Manager explicitly deletes the project.
- *"Do you support multi-currency?"* — Yes — `currency_code CHAR(3)` on `projects` and `invoices`. Today the prototype assumes USD; the schema is already future-proofed.
